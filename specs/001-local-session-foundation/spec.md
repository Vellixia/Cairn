# Feature Specification: Local Session Foundation

**Feature Branch**: `001-local-session-foundation`

**Created**: 2026-07-16

**Status**: Draft

**Input**: User description: "Build the local session foundation for Cairn: register a local Git repository, inspect exact repository state, create deterministic repository snapshots, and manage local agent sessions bound to exact repository state — fully offline, with append-only local event history, file-change observation, privacy-safe collection, a CLI, and robust failure handling."

## Clarifications

### Session 2026-07-16

- Q: How should Cairn identify the "agent process" for the single-active-session uniqueness rule (FR-017)? → A: Stable per-agent-instance UUID scoped to the repository, generated by Cairn's launcher/MCP adapter/hook/plugin when the agent starts and passed with every session request (e.g., via environment variable). Stable across daemon reconnects; expires when the agent instance ends. OS PID stored only as supporting liveness metadata. No fallback to agent type (two instances of the same agent in one repository must not collide).
- Q: What happens when a session start collides with an existing automatically active session for the same agent instance + repository? → A: Idempotent when healthy, recover/replace when stale. If the existing session has a recent liveness heartbeat, return it unchanged. If it has expired, its process is no longer alive, or recovery checks fail, mark it interrupted with a recorded event and start a new session from the current repository snapshot. *(Amended 2026-07-16, analysis A1: staleness is defined by resume-token lease expiry; a missing or unverifiable process ID never implies death — see FR-034.)*
- Q: What is the recovery policy for sessions that were active when the daemon stopped? → A: Recovering state with authenticated reattachment. On restart, previously active sessions move to a `recovering` state (never directly back to active). The same session resumes only when the same agent instance reconnects within a grace period presenting its stable instance ID and resume token; a session-recovered event is recorded and a fresh repository snapshot is captured. If no valid reattachment occurs before the grace period expires, the session is marked interrupted. Repository changes during daemon downtime refresh the snapshot, never auto-terminate the session. *(Amended 2026-07-16, analysis I3: a wrong or missing resume token rejects only that reattachment request — recorded as an audit event without token values — and never mutates the recovering session; interruption occurs only at grace-deadline expiry or authenticated owner stop.)*
- Q: What should inspection output contain for ignored files, given ignored sets can exceed 100k paths? → A: Structured summary with paginated drill-down. Default inspection reports total ignored count, ignore-rule source, collapsed top-level ignored roots, bounded samples, and a truncated indicator. Large ignored directories (e.g., node_modules/, target/) are represented as collapsed subtrees, never enumerated per descendant. Full details available through a separate filtered, paginated command or stream — never one unbounded response.
- Q: What anchors the stable repository identity across directory moves/renames and duplicate clones? → A: Git-private UUID with separate repository and worktree identities. A generated UUID is stored inside Git's private metadata (repository ID under the Git common directory; a distinct worktree ID under each linked worktree's Git directory), never in the tracked working tree. The local database stores the current canonical path as mutable metadata. Moves/renames preserve identity; a fresh clone gets new IDs (Git-private metadata is not transferred). If a directory is manually copied including its Git data, Cairn detects the same ID live at multiple paths, assigns a new identity to the copy, and records the relationship.
- Q: What does "show the active session" return when multiple sessions are active in one repository? → A: Adaptive cardinality with explicit ambiguity. With a trusted agent instance identifier in context, return only that instance's active or recovering session. Without agent context: exactly one session → return it directly; multiple → return a compact session list explicitly marked ambiguous. Selectors supported (session ID, agent instance, agent type). Never silently choose one session by recency.

### Session 2026-07-19

- Q: When may `session start` report success relative to filesystem-watcher startup? → A: Only after the operating-system watcher is installed, its event-processing path has acknowledged readiness, and an authoritative Git reconciliation has completed after installation. The required order is session creation → watcher installation request → watcher-ready acknowledgement → post-install Git reconciliation → session-start response. The reconciliation closes the installation window by detecting changes made after the initial snapshot but before watcher readiness.
- Q: What happens if watcher installation or the post-install reconciliation fails? → A: The start request returns a stable machine-readable `WATCHER_START_FAILED` error, never a success result. Its typed, schema-constrained payload discriminates watcher failure with `kind=watcher_start_failure` and identifies `stage=install|reconcile`; it is not arbitrary JSON and never exposes raw internal errors, paths, environment values, tokens, repository contents, or secrets. A newly created session must not remain falsely healthy: it transitions to `interrupted` and the failure is recorded through the append-only `session.interrupted` event model. Filesystem notifications remain advisory; Git reconciliation remains authoritative.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Register a repository and inspect its exact state (Priority: P1)

A developer working in an existing Git repository initializes Cairn once. Cairn
detects the repository root, records a stable local repository identity, and can
then report the exact repository state on demand: repository root, current
branch or detached-HEAD state, HEAD commit, default remote (when one exists),
staged changes, unstaged changes, untracked files, ignored files, and active
Git worktree information.

**Why this priority**: Every other capability (snapshots, sessions, events)
depends on a registered repository and accurate state inspection. This story
alone already delivers value: a trustworthy, machine-readable answer to "what
exact state is this repository in right now?"

**Independent Test**: In a fresh Git repository, run initialization and then
state inspection. Verify the reported root, branch, HEAD, and change lists
match Git's own view. Run initialization again and verify no duplicate
registration. Run initialization in a non-Git directory and verify a clear
error with no partial registration left behind.

**Acceptance Scenarios**:

1. **Given** an existing Git repository not yet registered, **When** the user initializes Cairn inside it, **Then** Cairn detects the repository root, persists a stable local repository identity, and confirms registration.
2. **Given** a repository already registered, **When** the user re-runs initialization, **Then** Cairn reports the existing registration and creates no duplicate.
3. **Given** a directory that is not part of any Git repository, **When** the user initializes Cairn, **Then** Cairn produces a clear error and persists nothing.
4. **Given** a registered repository with staged, unstaged, and untracked changes, **When** the user inspects repository state, **Then** the output correctly distinguishes staged changes, unstaged changes, untracked files, and ignored files, and reports branch (or detached HEAD), HEAD commit, default remote when present, and active worktree information.
5. **Given** no network connectivity, **When** the user registers a repository and inspects state, **Then** both operations succeed entirely locally.

---

### User Story 2 - Start, inspect, and stop a session bound to an exact repository state (Priority: P2)

A developer starts a Cairn session for a named agent. Cairn captures a
deterministic snapshot of the exact repository state at session start, binds
the session to it, and lets the developer inspect the active session (identity,
agent, start snapshot, current snapshot, start time, state) and stop it.

**Why this priority**: This is the core promise of the feature — an agent
session anchored to an exact repository state, not just a branch name. It
requires User Story 1 but delivers the primary user value.

**Independent Test**: In a registered repository, start a session for an agent,
verify the session shows a start snapshot whose fingerprint changes if and only
if relevant repository state changes, then stop the session and verify its
final state is recorded.

**Acceptance Scenarios**:

1. **Given** a registered repository, **When** the user starts a session for a named agent, **Then** Cairn creates a snapshot of the current repository state and records a session with a stable identifier, local user identity, repository, agent type, start snapshot, current snapshot, start time, and state.
2. **Given** an active session, **When** the user inspects the active session, **Then** all session attributes above are shown, including the start snapshot and the current snapshot.
3. **Given** an active session, **When** the user stops it, **Then** the session transitions to a stopped state and a session-stop event is recorded.
4. **Given** an active, healthy session (unexpired resume-token lease) for agent instance A in a repository, **When** the same agent instance requests another automatic session start in the same repository, **Then** Cairn returns the existing session unchanged (idempotent start) — never a second active session for that agent-instance-and-repository combination.
5. **Given** an active session for agent instance A whose lease has expired without reattachment (or whose recorded process is verifiably dead), **When** the same agent instance requests a session start, **Then** Cairn marks the stale session interrupted with a recorded event (including the liveness reason code) and starts a new session from the current repository snapshot.
6. **Given** an active session for one agent instance in a repository, **When** a different agent instance — including a second instance of the same agent type — starts a session in the same repository, **Then** both sessions coexist independently.
7. **Given** an unchanged repository state, **When** snapshots are taken repeatedly, **Then** every snapshot produces the identical fingerprint.
8. **Given** any relevant repository-state change (new commit, branch switch, staged edit, unstaged edit, untracked file added or deleted), **When** a new snapshot is taken, **Then** the snapshot fingerprint differs from the previous one.
9. **Given** a session-start request, **When** the session is created, **Then** Cairn does not return success until the worktree watcher is installed, its event-processing path is ready, and a post-install Git reconciliation has completed.
10. **Given** a file change made after the initial session snapshot but before watcher installation completes, **When** the post-install reconciliation runs, **Then** the returned session's current snapshot reflects that change.
11. **Given** watcher installation or post-install reconciliation fails, **When** session start completes, **Then** Cairn returns `WATCHER_START_FAILED` with a schema-constrained `watcher_start_failure` payload and the correct `install` or `reconcile` stage, records an append-only interruption event, leaves no falsely healthy active session, and leaks no raw internal details, paths, environment values, repository contents, or tokens.

### User Story 3 - Track repository state continuously during an active session (Priority: P3)

While a session is active, Cairn observes filesystem changes, reconciles them
against actual Git state, and updates the session's current snapshot so that at
any moment the developer (or the agent) can ask: "what state did this session
start from, and what state is it in now?"

**Why this priority**: Continuous tracking turns a point-in-time anchor into a
live record. Valuable, but sessions and snapshots must exist first.

**Independent Test**: Start a session, modify files (create, edit, delete,
switch branch), and verify the current snapshot updates to reflect the new
state, that branch changes are recorded as events, and that rapid bursts of
file changes produce a correct final state.

**Acceptance Scenarios**:

1. **Given** an active session, **When** a tracked file is modified, **Then** Cairn updates the session's current snapshot to reflect the change within a short, bounded delay.
2. **Given** an active session, **When** the user switches branches, **Then** Cairn records a branch-change event and the current snapshot reflects the new branch and HEAD.
3. **Given** an active session, **When** many filesystem events occur in rapid succession, **Then** Cairn may coalesce intermediate updates but the final recorded state matches the actual repository state.
4. **Given** a filesystem notification that does not correspond to a real repository-state change, **When** Cairn reconciles against Git state, **Then** no new authoritative snapshot is created.
5. **Given** an active session, **When** a file is deleted or a Git rebase rewrites history, **Then** the current snapshot and events reflect the resulting real state without corruption.
6. **Given** a session whose watcher is being installed, **When** files are created, modified, renamed, or deleted during that installation window, **Then** the post-install reconciliation records the final authoritative Git state without losing any final change. Deletion acceptance MUST include a file present in the initial authoritative snapshot and deleted while installation is barrier-paused, then prove the returned/current snapshot and repository-change event reflect the deletion without a duplicate logical change event.
7. **Given** an active session and a dropped filesystem notification, **When** the reconciliation path is explicitly triggered, **Then** Cairn converges to actual Git state without relying on the missing notification.
8. **Given** a notification and the post-install or periodic reconciliation observe the same repository change, **When** both paths complete, **Then** exactly one authoritative repository-state change is recorded for that snapshot transition.
9. **Given** the daemon restarts with a recovering session, **When** its watcher is reinstalled, **Then** readiness is acknowledged and an authoritative reconciliation completes before tracking is treated as healthy.

---

### User Story 4 - Survive restarts with durable history and session recovery (Priority: P4)

A developer's daemon restarts (crash, reboot, upgrade) while a session is
active. After restart, Cairn still has all committed events — repository
registration, snapshot creation, session start, repository-state changes,
branch changes, session stops — and the developer can recover the interrupted
session or see it correctly marked as interrupted.

**Why this priority**: Durability is what makes the recorded history
trustworthy evidence rather than best-effort telemetry. It hardens Stories 1–3.

**Independent Test**: Start a session, generate events, kill the daemon
process, restart it, and verify no committed events were lost, the session is
recoverable or correctly marked interrupted, and new snapshots reconcile
against the actual current repository state.

**Acceptance Scenarios**:

1. **Given** committed local events, **When** the daemon restarts, **Then** all committed events remain readable and unmodified.
2. **Given** an active session at the time of an unclean daemon stop, **When** the daemon restarts, **Then** the session enters a `recovering` state; it resumes only when the same agent instance reattaches before the grace deadline with matching instance identifier and resume token, recording a session-recovered event and a fresh snapshot. A mismatched token rejects only that request (audit event, session stays recovering); the session is marked interrupted only at grace-deadline expiry — anchored to when it entered recovering, unchanged by further restarts — or authenticated stop. It is never silently lost.
3. **Given** corrupted or unavailable local state, **When** Cairn starts, **Then** it reports the problem clearly and does not fabricate repository or session state.
4. **Given** repository changes made while the daemon was down, **When** the daemon restarts and a snapshot is requested, **Then** the snapshot reflects the actual current repository state.

### Edge Cases

- Detached HEAD: inspection and snapshots must represent detached-HEAD state explicitly instead of reporting a branch name.
- Repository without any remote: registration, inspection, and snapshots work; remote fields are absent, not erroneous.
- Branch switch or Git rebase during an active session: events recorded; snapshots reflect post-change reality; no stale state persists as current.
- Dirty working tree at session start: start snapshot captures staged, unstaged, and untracked fingerprints — dirty state is first-class, not an error.
- File deletion (tracked or untracked): reflected in fingerprints and current snapshot.
- Concurrent repository changes while a snapshot is being generated: the snapshot must be internally consistent (represent one coherent state) or be retried; a torn half-state must never be recorded as authoritative.
- Daemon restart mid-session: see User Story 4.
- Corrupted or unavailable local storage: clear error; no partial writes presented as valid history.
- Non-Git directory initialization: clear error, zero persisted artifacts.
- Duplicate initialization: idempotent, no duplicate identity.
- Repository using multiple Git worktrees: each working tree's state is inspected and snapshotted for the worktree in which Cairn operates; each linked worktree carries its own worktree identity; worktree information is reported.
- Repository directory moved or renamed: identity persists (Git-private identifier); canonical path metadata is updated.
- Directory manually copied including Git data: same identity detected at two live paths → the copy receives a new identity and the relationship is recorded; sessions and events never silently merge across the two.
- Ignored files containing secrets: names/paths may be observed for state purposes per exclusion rules, but secret values and ignored-file contents are never persisted.
- Enormous ignored sets (e.g., 100k+ paths under node_modules/ or target/): inspection stays fast and bounded — summary with collapsed roots and truncation indicator, drill-down only via pagination.
- Multiple concurrently active sessions in one repository: session inspection without agent context returns an explicitly ambiguous list — never a silent recency-based pick.
- Bare repository (`git init --bare`): initialization is rejected with a stable not-a-worktree error and a clear message; no identity markers, database records, or events are created.
- Deleted Cairn identity markers (e.g., `rm -rf .git/cairn/`): re-initialization searches local records by normalized canonical path and Git common-directory metadata; exactly one compatible match restores the identity (with a recorded restoration event); no unique match creates a new identity only after an explicit warning (human output) and explicit status (machine output). Identity is never inferred from remote URL alone.
- File changes during watcher installation: session start must not expose a readiness gap; the post-install Git reconciliation captures the final state before success is returned.
- Watcher installation or reconciliation failure: the start request fails with stable machine-readable data, the created session is interrupted through the append-only event/state model, and no active session is presented as healthy.
- Dropped or duplicated filesystem notifications: a dropped hint cannot prevent eventual authoritative reconciliation, and duplicate hints cannot create duplicate repository-state events for the same snapshot transition.

## Requirements *(mandatory)*

### Functional Requirements

**Repository registration**

- **FR-001**: Users MUST be able to initialize Cairn from any directory inside an existing Git repository; Cairn MUST detect the repository root automatically.
- **FR-002**: Cairn MUST persist a stable local repository identity anchored in a generated identifier stored in Git-private metadata (repository identity in the Git common directory; a separate worktree identity in each linked worktree's private Git directory) — never in the tracked working tree. Identity MUST survive restarts, directory moves/renames, and changes to branch, HEAD, or working tree. The current canonical path is stored as mutable metadata alongside the identity. A fresh clone MUST receive a new identity. If the same identity is detected live at multiple paths (e.g., a manual directory copy including Git data), Cairn MUST assign a new identity to the copied instance and record the relationship between the two.
- **FR-003**: Re-running initialization on an already-registered repository MUST NOT create a duplicate registration and MUST report the existing registration.
- **FR-004**: Initializing in a directory that is not part of a Git repository MUST fail with a clear error and MUST leave no partial registration data.
- **FR-005**: Repository registration MUST complete without any network connectivity or central-server availability.

**Repository-state inspection**

- **FR-006**: Cairn MUST report, on demand: repository root, current branch or explicit detached-HEAD state, HEAD commit identifier, default remote when one exists, staged file changes, unstaged file changes, untracked files, ignored files, and active Git worktree information.
- **FR-007**: Cairn MUST NOT treat a branch name alone as an exact repository state; exact state MUST incorporate HEAD commit and working-tree fingerprints.
- **FR-035**: Ignored files in inspection output MUST be reported as a structured summary: total ignored count, ignore-rule source, collapsed top-level ignored roots, bounded samples, and an explicit truncation indicator. Large ignored directories MUST be represented as collapsed subtrees rather than enumerating every descendant. Full ignored-file detail MUST be available only through a separate filtered, paginated command or stream — never as one unbounded response.

**Deterministic snapshots**

- **FR-008**: A snapshot MUST record: current branch (or detached-HEAD marker), HEAD commit, staged-change fingerprint, unstaged-change fingerprint, untracked-file fingerprint, and a final snapshot fingerprint derived from those components.
- **FR-009**: Two snapshots of the same repository state MUST produce identical snapshot fingerprints (determinism).
- **FR-010**: Any relevant repository-state change (commit, branch switch, staged change, unstaged change, untracked file added/removed/modified/deleted) MUST produce a different snapshot fingerprint.
- **FR-011**: Snapshots MUST store metadata and hashes only; they MUST NOT duplicate repository contents.
- **FR-012**: A snapshot generated while the repository is concurrently changing MUST represent one internally consistent state or be retried; an inconsistent snapshot MUST NOT be recorded as authoritative.

**Session lifecycle**

- **FR-013**: Users MUST be able to start a local session for a named agent type in a registered repository.
- **FR-014**: A session MUST be associated with: local user identity, repository, agent type, start snapshot, current snapshot, start time, session state, and a stable session identifier.
- **FR-015**: Users MUST be able to inspect the active session and see all attributes in FR-014.
- **FR-016**: Users MUST be able to stop an active session; the stop MUST be recorded.
- **FR-017**: At most one automatically active session MUST exist per agent instance and repository combination. An agent instance is identified by a stable per-instance identifier (UUID) generated when the agent starts (by Cairn's launcher, adapter, hook, or plugin) and supplied with every session request; the identifier remains stable across daemon reconnects and expires when the agent instance ends. The OS process ID is recorded only as supporting liveness metadata. Sessions for different agent instances — including two instances of the same agent type — in the same repository MUST coexist independently.
- **FR-018**: After a daemon restart, sessions that were active MUST move to a `recovering` state — never silently disappear and never return directly to active. When a session first enters `recovering`, Cairn MUST persist `recovering_since`; the grace deadline is `recovering_since` plus the configured grace period, and subsequent daemon restarts MUST preserve the existing timestamp and MUST NOT extend the deadline. A recovering session MUST resume (with a recorded session-recovered event and a fresh repository snapshot) only when the same agent instance reattaches before the deadline presenting its stable instance identifier and resume token. A wrong or missing resume token MUST reject only that reattachment request — recorded as an audit event that never contains token values — and MUST NOT interrupt, stop, or otherwise mutate the recovering session. The session becomes `interrupted` only when the grace deadline expires or an authenticated owner explicitly stops it. Repository changes that occurred while the daemon was down MUST refresh the snapshot and MUST NOT by themselves terminate the session.
- **FR-034**: A session-start request that collides with an existing automatically active session for the same agent instance and repository MUST be idempotent when the existing session is healthy (unexpired resume-token lease): the existing session is returned unchanged. A session is stale only when its lease has expired without valid reattachment; a verifiably dead recorded process (`agent_pid`) MAY confirm staleness earlier, but a missing or unverifiable process ID MUST NOT be treated as evidence of death. Liveness determinations MUST record a reason code (heartbeat_expired, process_dead, reattach_timeout, process_unknown). Callers that cannot heartbeat immediately MUST receive a longer, configurable initial lease. When the existing session is stale, Cairn MUST mark it interrupted with a recorded event and start a new session from the current repository snapshot.

**Local event history**

- **FR-019**: Cairn MUST record append-only local events for: repository registration, snapshot creation, session start, repository-state change, branch change, session stop, session interruption, session recovery, rejected session reattachment (audit, never containing token values), and identity-marker restoration.
- **FR-020**: Committed events MUST NOT be modified or deleted by normal operation, and MUST survive daemon restart.

**File-change observation**

- **FR-021**: While a session is active, Cairn MUST observe filesystem changes in the repository and update its understanding of repository state.
- **FR-022**: Filesystem notifications MUST be treated as advisory; Cairn MUST reconcile against actual Git state before recording an authoritative snapshot.
- **FR-023**: Rapid repeated filesystem events MAY be coalesced, but the final recorded state MUST match the actual repository state (no lost final state).
- **FR-037**: A successful session-start response MUST NOT be returned until the worktree watcher has been installed with the operating system, the watcher event-processing path has acknowledged readiness, and an authoritative Git reconciliation has completed after installation. The required order is session creation → watcher installation request → watcher-ready acknowledgement → post-install Git reconciliation → session-start response. The post-install reconciliation MUST detect relevant changes made between the initial snapshot and completion of watcher installation.
- **FR-038**: If watcher installation or post-install reconciliation fails, Cairn MUST NOT return a successful session-start result and MUST NOT leave a falsely healthy active session. A newly created session MUST transition to `interrupted`; the failure MUST be recorded through the append-only `session.interrupted` event model; and the request MUST return the stable `WATCHER_START_FAILED` machine-readable error. Its typed payload MUST be schema-discriminated by the error code, MUST identify `kind=watcher_start_failure` and `stage=install|reconcile`, and MUST NOT accept unconstrained arbitrary JSON. IPC and CLI JSON contracts MUST cover both stages, both CLI paths MUST exit 1, schema-breaking changes MUST trip contract tests, and raw internal errors, paths, environment values, repository contents, or tokens MUST NOT leak. Filesystem notifications remain advisory and Git reconciliation remains authoritative.

**Offline operation**

- **FR-024**: All capabilities in this feature MUST work with no network connectivity.
- **FR-025**: All session, snapshot, repository, and event data required by this feature MUST be stored locally and remain available after restart.

**Privacy and exclusions**

- **FR-026**: Cairn MUST respect Git ignore rules and a Cairn-specific ignore file when observing and fingerprinting repository state.
- **FR-027**: Cairn MUST NOT persist secret values or entire ignored-file contents in order to create a snapshot.
- **FR-028**: No file contents may be uploaded anywhere as part of this feature.

**CLI experience**

- **FR-029**: Cairn MUST provide user-invocable commands equivalent to: initialize in current repository, inspect repository state, start a session for a named agent, show the active session, send a session heartbeat, reattach to a recovering session, stop a session, and display local daemon status. Heartbeat and reattach require the session identifier, agent instance identifier, and resume token. Resume tokens MUST NOT appear in human-readable output, logs, or ordinary command-line arguments (visible in process listings); secure input via stdin, inherited environment, or file mechanism MUST be supported, and machine-readable output delivers tokens only where the contract explicitly issues them.
- **FR-030**: Every command MUST offer human-readable output and a stable machine-readable output mode suitable for scripting and agent consumption.
- **FR-031**: Command failures MUST produce clear, actionable error messages and distinct failure signals in machine-readable mode.
- **FR-036**: The show-active-session command MUST resolve adaptively: when a trusted agent instance identifier is present in the calling context, return only that instance's active or recovering session; otherwise return the single session directly when exactly one exists, and when multiple exist return a compact session list explicitly marked ambiguous. Selection by session identifier, agent instance, or agent type MUST be supported. Cairn MUST NOT silently choose one session based on recency.

**Failure handling**

- **FR-032**: Cairn MUST behave correctly under: detached HEAD, repositories without remotes, branch switches, Git rebases, dirty working trees, file deletion, daemon restart, corrupted or unavailable local state, and concurrent state changes during snapshot generation.
- **FR-033**: On corrupted or unavailable local state, Cairn MUST report the condition clearly and MUST NOT present fabricated or partial state as valid.

### Key Entities

- **Repository**: A registered local Git repository; carries a stable local identity (generated identifier in Git-private metadata under the Git common directory), a per-worktree identity for linked worktrees, the current canonical root path (mutable metadata), and default remote (when present). Identity is independent of branch, working-tree state, and directory location.
- **Repository State (inspection result)**: A point-in-time, non-persisted view: root, branch or detached HEAD, HEAD commit, remote, staged/unstaged/untracked/ignored file sets, worktree information.
- **Snapshot**: A persisted, deterministic fingerprint of one exact repository state: branch, HEAD commit, staged/unstaged/untracked fingerprints, and final combined fingerprint. Metadata and hashes only.
- **Session**: A unit of agent work bound to a repository: stable identifier, local user identity, repository, agent type, agent instance identifier (per-instance UUID; optional `agent_pid` kept only as supporting liveness metadata), resume-token lease (stored internally only as `resume_token_hash`, with a lease-expiry timestamp), `recovering_since` (persisted when the session enters recovery), start snapshot, current snapshot, start time, state (active, recovering, stopped, interrupted).
- **Event**: An append-only, timestamped record of one occurrence (registration, snapshot creation, session start/stop, repository-state change, branch change) with references to the entities involved.
- **Local User Identity**: The operating-system-level identity of the developer on this machine; used to attribute sessions without any account system.
- **Cairn Ignore File**: A repository-local exclusion list, layered on top of Git ignore rules, controlling what Cairn may observe and fingerprint.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: With all external network access disabled by an OS-level network namespace or container isolation mechanism that still permits local IPC and filesystem access, a developer can register a repository, start a session, change repository state, and retrieve both the session's start state and current state — 100% of required data served from local storage. Acceptance evidence MUST name the isolation mechanism; `cargo --offline` alone is insufficient.
- **SC-002**: Across at least 100 repeated snapshot operations on an unchanged repository state, 100% of snapshots produce the identical fingerprint.
- **SC-003**: Each relevant repository-state change (commit, branch switch, stage, edit, untracked add, file delete) produces a changed snapshot fingerprint in 100% of test cases; the session's current snapshot reflects the change within 5 seconds of filesystem quiescence. This includes changes made during watcher installation and immediately after session start returns; correctness tests MUST use synchronization barriers or explicit acknowledgements rather than timing sleeps as their primary mechanism.
- **SC-004**: Initialization in a non-Git directory leaves zero persisted artifacts in 100% of attempts, and re-initialization of a registered repository yields exactly one registration.
- **SC-005**: After exactly 100 forced daemon terminations at arbitrary points in the acceptance execution, zero committed events are lost, and 100% of sessions active at termination enter the recovering state and are subsequently resumed (on valid reattachment) or explicitly marked interrupted (on grace-period expiry) — never silently lost. The harness MAY retain a smaller configurable default for ordinary local runs, but an 8- or 20-iteration run does not satisfy this criterion; CI and evidence MUST report all 100 completed iterations.
- **SC-006**: An audit of all persisted data after snapshotting a repository containing ignored secret files finds zero secret values and zero ignored-file contents.
- **SC-007**: Repository state inspection and snapshot creation each complete in under 2 seconds on a repository with 10,000 tracked files on typical developer hardware. Acceptance requires explicit execution of `cargo test -p cairn-daemon --test perf -- --ignored` on the exact frozen implementation commit; an ordinary workspace run that reports the performance test as ignored is insufficient. Evidence MUST record the implementation SHA, OS/architecture, tracked-file fixture size, measured inspect and snapshot durations, the limits, and pass/fail result.
- **SC-008**: A script or agent consuming machine-readable output can parse 100% of command results for every provided command (eight in this feature, including heartbeat and reattach) without output-format ambiguity across repeated runs.

### Completion Gate

## Assumptions

- Local user identity is derived from the operating-system user; no login, accounts, or central identity are involved in this feature.
- "Ignored files" in inspection output means summary metadata and paths as permitted by exclusion rules (per FR-035) — never their contents.
- Fingerprints are derived from content hashes and Git metadata; the exact hashing scheme is an implementation choice, constrained only by determinism (FR-009) and sensitivity (FR-010).
- The coalescing window for filesystem events is an implementation choice, constrained by the 5-second quiescence bound in SC-003 and the no-lost-final-state rule (FR-023).
- "Agent type" is a caller-supplied name (e.g., the agent product name); this feature does not validate agents or integrate agent plugins. The agent instance identifier is distinct from agent type: a per-instance UUID generated at agent startup and conveyed with session requests (e.g., via environment variable).
- When a repository uses multiple Git worktrees, Cairn operates per working tree; the shared repository identity lives in the Git common directory while each linked worktree carries its own worktree identity, and worktree information is reported accurately.
- Session identifiers must be stable and unique per machine; global uniqueness across machines is not required by this feature.
- The reattachment grace period and heartbeat interval are implementation choices; both must be long enough to survive routine daemon restarts and short enough that stale sessions are detected promptly.
- Timestamps are recorded in a consistent, timezone-unambiguous form.
- The daemon and CLI run on the developer's machine with ordinary user privileges; no elevated permissions are required.
