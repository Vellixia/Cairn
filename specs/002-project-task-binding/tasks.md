# Tasks: Project and Task Binding Foundation

**Input**: Authoritative design documents from `specs/002-project-task-binding/`:
`spec.md`, `plan.md`, `research.md`, `data-model.md`, `migration-design.md`,
`event-catalog.md`, `module-ownership.md`, `compatibility-matrix.md`,
`testing-evidence.md`, `quickstart.md`, and `contracts/`.

**Tests**: Required by the specification. Story tests are written before their
implementation tasks and must initially fail for the missing behavior.

**Organization**: Tasks preserve Feature 001 first, establish shared domain/storage
foundations, deliver User Stories 1–5 in dependency order, then collect exact-commit
evidence and run the final convergence gate.

## Format: `[ID] [P?] [Story?] Description`

- **[P]**: Safe to execute concurrently after its stated prerequisites because it uses
  different files and does not depend on another incomplete parallel task.
- **[US1]…[US5]**: Maps to the prioritized user stories in `spec.md`.
- Every task cites its requirement, scenario, success criterion, plan decision, or
  constitution principle.
- A checked box is never evidence by itself; the named test, migration, command, or
  completed workflow must pass.

## Dependencies and Execution Order

### Phase dependencies

1. **Phase 1** has no implementation dependency; T001–T007 are a hard compatibility
   gate before T008–T009.
2. **Phase 2** depends on Phase 1 and defines types consumed everywhere else.
3. **Phase 3** depends on Phase 2 and blocks every user-story mutation.
4. **US1 / Phase 4** depends on Phase 3 and creates project/repository scope.
5. **US2 / Phase 5** depends on US1 because tasks belong to active projects.
6. **US3 / Phase 6** depends on US1 + US2 + Feature 001 sessions.
7. **US4 / Phase 7** depends on US3's shared binding transaction and extends the existing
   start path.
8. **Phase 8** depends on US1–US4 method implementations.
9. **US5 / Phase 9** depends on Phases 3–8 and verifies the combined ledger/migration.
10. **Phase 10** depends on all implementation and local acceptance tasks through T113;
    T117–T121 depend on the immutable T116 SHA.

```text
T001–T007 baseline
→ T008–T022 workspace/domain/protocol
```

## Implementation Strategy

### MVP first

The smallest independently useful Feature 002 slice is Phases 1–4:

1. preserve and freeze Feature 001 behavior;
2. establish typed/storage foundations;
3. migrate safely;
4. deliver project creation and repository association (US1);
5. stop and run the US1 independent test before adding task or binding behavior.

### Incremental delivery

1. **US1**: project and repository scope.
2. **US2**: immutable task-revision and goal-contract scope.
3. **US3**: explicit one-time binding for existing sessions.
4. **US4**: correctly scoped new sessions through the unchanged start lifecycle.
5. **US5**: combined migration/replay/offline proof.
6. Freeze one implementation SHA, collect real platform evidence, then converge.

## Scope guardrails

No task authorizes central synchronization, PostgreSQL, user accounts/membership,
roles, repository transfer/removal, cross-device continuity, MCP, context compilation,
AI memory, truth claims, embeddings, drift detection, product completion verification,
web UI, multi-agent orchestration, session unbind/rebind/revision transition, or
Feature 003 implementation.
