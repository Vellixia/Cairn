# Cairn v0.8.0 — Implementation Plan

> **Goal:** Scoped memory (Session / Project / Global), SurrealDB replacing HelixDB+MinIO,
> auto project detection, LLM intelligence layer, RAG document store, session TTL + cron —
> plus an **autonomy layer** (S8–S9) so Cairn runs unattended day-to-day: memories auto-promote
> with undo instead of sitting in an approval queue, safe drift auto-approves while dangerous
> drift still waits for a human, workspace docs auto-sync into RAG, and the system self-heals
> (offline spool, cron watchdog, LLM budget cap) without losing any existing functionality.

---

## How to Read This Plan

Each task has:
- **What:** what you are building
- **Why:** why it is needed
- **Files:** exact files to create or edit
- **Changes:** exact struct fields, function signatures, and code to write
- **Done when:** specific verifiable condition

Work the sprints in order (S1 → S9). Each sprint produces a working, passing build.
Never start S2 before S1 compiles. Run `cargo check --workspace` after every sprint.

S1–S7 build the core migration (SurrealDB, scopes, auto project detection, session TTL + cron,
LLM intelligence, RAG, quality/dashboard). **S8–S9 are the autonomy layer**: they remove the
standing daily chores (approval queues, manual drift triage, manual doc ingestion) by replacing
them with act-by-default behavior, an observable log, cheap undo, and self-correcting safety
nets — the goal being Cairn needs far less day-to-day human interaction while every existing
function still works exactly as before for anyone who sets the relevant `*_AUTOPILOT=off` /
`CAIRN_SELFTUNE=off` env var.

### S1.4 — Create `cairn-store/src/surreal.rs`

**File:** `crates/cairn-store/src/surreal.rs` (new file)

This implements the `StoreBackend` trait (same interface as the old helix backend).
The `StoreBackend` trait is defined in `crates/cairn-store/src/lib.rs` — do not change it.

**Cargo.toml for `cairn-store`** — add to `[dependencies]`:
```toml
surrealdb = { workspace = true }
```

**Key functions to implement:**
```rust
use surrealdb::{Surreal, engine::any::Any};
use crate::StoreBackend;

pub struct SurrealStore {
    db: Surreal<Any>,
}

impl SurrealStore {
    /// Connect and apply schema. Called once at server startup.
    pub async fn open(url: &str, user: &str, pass: &str) -> Result<Self> {
        let db = surrealdb::engine::any::connect(url).await?;
        db.signin(surrealdb::opt::auth::Root { username: user, password: pass }).await?;
        db.use_ns("cairn").use_db("main").await?;
        db.query(include_str!("schema.surql")).await?.check()?;
        Ok(Self { db })
    }
}

impl StoreBackend for SurrealStore {
    // Implement each method by translating the old HelixDB calls to SurrealQL.
    // The trait method signatures do NOT change — only the implementation body changes.
}
```

For each `StoreBackend` method, map the HelixDB operation to SurrealDB:
- `store(memory)` → `db.create("memory").content(record).await?`
- `get(id)` → `db.select(("memory", id)).await?`
- `update(id, patch)` → `db.update(("memory", id)).merge(patch).await?`
- `delete(id)` → `db.delete(("memory", id)).await?`
- `list(filter)` → `db.query("SELECT * FROM memory WHERE ...").await?`
- `search_bm25(q)` → `db.query("SELECT *, search::score(1) AS score FROM memory WHERE content @1@ $q ORDER BY score DESC LIMIT $n").bind(("q", q)).await?`
- `search_vector(embedding, n)` → `db.query("SELECT *, vector::similarity::cosine(embedding, $vec) AS score FROM memory ORDER BY score DESC LIMIT $n").bind(("vec", embedding)).await?`

**HNSW index:** Create it after the first embedding is stored (we need the dimension):
```rust
pub async fn ensure_hnsw_index(&self, dim: usize) -> Result<()> {
    self.db.query(format!(
        "DEFINE INDEX IF NOT EXISTS idx_embedding ON memory \
         FIELDS embedding HNSW DIMENSION {dim} DIST COSINE TYPE F32;"
    )).await?.check()?;
    // Same for chunk table
    self.db.query(format!(
        "DEFINE INDEX IF NOT EXISTS idx_chunk_emb ON chunk \
         FIELDS embedding HNSW DIMENSION {dim} DIST COSINE TYPE F32;"
    )).await?.check()?;
    Ok(())
}
```

```rust
fn check_database(cfg: &cairn_core::Config) -> Check {
    // Attempt a simple HTTP/WS ping to CAIRN_DB_URL.
    // In remote mode (CAIRN_SERVER set) → skip (server handles DB).
    if std::env::var_os("CAIRN_SERVER").is_some() {
        return Check { name: "database", ok: true, detail: "(skipped — remote mode)".into() };

    // Try: GET <db_url>/health (SurrealDB exposes /health on HTTP)
    let health_url = cfg.db_url
        .replace("ws://", "http://")
        .replace("wss://", "https://");
    match ureq::get(&format!("{health_url}/health")).call() {

            detail: format!("{} (surreal ok)", cfg.db_url),

        Ok(r) => Check {
            name: "database", ok: false,
            detail: format!("{} (HTTP {})", cfg.db_url, r.status()),
        },
        Err(e) => Check {
            name: "database", ok: false,
            detail: format!("{} (unreachable: {e})", cfg.db_url),
        },
    }
}
```

## Sprint 2 — Scope Model

**Goal:** Every memory has a `scope_type` + `scope_id`. Recall always returns
Global + current Project + current Session memories, weighted by scope.

**Corrected against reality (2026-07-02, after Sprint 1 shipped):** the original draft of
this sprint assumed `cairn-memory` builds raw SurrealQL strings against the store and that
`crates/cairn-api/src/middleware/` already existed. Neither is true. `cairn-memory` only has
`cairn-store`'s synchronous, backend-agnostic `Store` API (`all_memories()`,
`semantic_recall()`, etc.) - no SQL access, by design, so the storage engine can keep
changing (as it just did) without touching memory/ranking logic. `recall_for_org()`
(`crates/cairn-memory/src/lib.rs:193`) already does exactly this shape of filtering for
`org_id`: pull `all_memories()`, filter in Rust, then run BM25 + `semantic_recall` + graph
proximity through an inline RRF-weighted fusion (lines 264-328, no separate `rrf_fuse()`
function). There is no `middleware/` directory in `cairn-api` at all - `security_headers`
(a flat module, wired via `.layer(middleware::from_fn(security_headers::security_headers))`
in `lib.rs`) is the closest existing analog and the pattern to copy. There is also no
`access_log` table or concept anywhere yet (only a per-memory `access_count` counter), and
`OrgId` itself isn't wired from HTTP into `MemoryEngine` today (call sites hardcode
`OrgId::default()`) - Sprint 2 is genuinely the first per-request scope context Cairn has.
The plan below reflects the corrected design; the sections after it were rewritten from
scratch, not lightly edited.

### S2.2 — `ScopeCtx` extractor in `cairn-api` (not `Config` — scope is per-request)

**File:** `crates/cairn-api/src/scope.rs` (new, flat module — mirrors `security_headers`,
not a `middleware/` subdirectory)

Project/session context is inherently per-HTTP-request, not a static server setting, so it
does not belong on `Config` (the original draft's S2.2 was wrong on this point too).

```rust
#[derive(Clone, Debug, Default)]
pub struct ScopeCtx {
    pub project_id: Option<String>,
    pub session_id: Option<String>,
}

pub async fn scope_middleware(headers: HeaderMap, mut req: Request, next: Next) -> Response {
    let project_id = headers.get("X-Cairn-Project").and_then(|v| v.to_str().ok()).map(str::to_string);
    let session_id = headers.get("X-Cairn-Session").and_then(|v| v.to_str().ok()).map(str::to_string);
    req.extensions_mut().insert(ScopeCtx { project_id, session_id });
    next.run(req).await
}
```

Wire it in `crates/cairn-api/src/lib.rs` next to the existing `security_headers` layer.
Handlers that need it take `Extension(ctx): Extension<ScopeCtx>` as an extractor argument
(standard axum `Extension` pattern - no new plumbing needed beyond the layer itself).

### S2.5 — Batched `access_log` writes

**File:** `crates/cairn-store/src/schema.surql` (new table), `crates/cairn-store/src/db.rs`
(new `StoreBackend` trait method with a default no-op impl - same pattern as

```rust
// db.rs trait addition (default no-op so memory_backend.rs doesn't need new logic):
fn record_access_batch(&self, entries: &[(String, Option<String>, Option<String>)]) -> Result<()> {
    let _ = entries;
    Ok(())
}
```

`surreal.rs` implements it as one multi-row `INSERT INTO access_log $rows` - a busy agent
recalling every turn can generate thousands of rows/hour, and one round-trip per recall
call (not one per returned memory) is what actually matters here; a background
`tokio::spawn` is unnecessary complexity for a single batched write and was dropped from
the original draft. The S4 `access-log-prune` cron job (`CAIRN_ACCESS_LOG_RETENTION_DAYS`,
default 90) keeps the table bounded regardless of write volume.

## Sprint 4 — Session TTL + Cron (shipped)

**Goal:** Sessions older than `CAIRN_SESSION_TTL_DAYS` auto-consolidate nightly.
A built-in scheduler runs background jobs.

**Corrected against reality before implementing** (same discipline as Sprints 2 and 3): the
draft assumed an entire parallel architecture that doesn't exist - an async `store.query(...)
.bind(...).await` raw-SQL surface, a `session` DB table, an `LlmClient` type, an `llm` field
on `AppState`, and two functions (`cairn_memory::run_llm_intelligence`,
`recompute_global_centroid`) that belong to the not-yet-built Sprint 5 LLM work. None of that
is buildable today without inventing Sprint 5 early. The real `StoreBackend`/`Store` surface
is deliberately synchronous with a fixed typed method set (no generic query API) - that's what
let Sprint 1 swap the storage engine without touching any other crate, and Sprint 4 keeps that
invariant instead of breaking it.

The scope was narrowed to what's buildable on real, already-shipped primitives:

- **"Session TTL"** is reinterpreted as *scope promotion*, not session-record bookkeeping -
  there is no `session` table to expire rows from. A `Session`-scoped memory
  (`cairn_core::ScopeType::Session`, from Sprint 2) that goes untouched (`updated_at`) past
  `CAIRN_SESSION_TTL_DAYS` is promoted straight to `Global`, so it doesn't stay silently
  walled off forever once its originating session is long gone. The `session_scope_default`
  config field and the "promote to project if it has one, else scope_default" branch from the
  draft were dropped - Sprint 2 never gave a `Session`-scoped memory a project association to
  fall back to, so `Global` is the only sound target.
- **"Memory decay"** drives the existing pure function
  [`cairn_memory::apply_decay`](../../../crates/cairn-memory/src/llm_consolidator.rs) (already
  shipped pre-v0.8.0) over every memory instead of a new decay algorithm. Persisting a decayed
  memory deliberately does **not** bump `updated_at` - `apply_decay` measures elapsed time
  *from* `updated_at`, so touching it on write would reset the decay clock every run and
  nothing would ever decay past the first period.
- **`llm-intelligence` and `global-centroid`** (the draft's Jobs 3 and 4) are deferred
  wholesale to Sprint 5, which is where an LLM client and a concept-centroid model will
  actually get built. Shipping stubs for functions that don't exist yet would just be dead
  code to delete later.
- **`access-log-prune`** ships as designed, as a new `StoreBackend` method rather than a raw
  `DELETE` query string.

`session_scope_default` and `cron_internal` (renamed `cron_enabled` - the draft's name
collided with nothing but `cron_enabled` reads more clearly at the call site) were adjusted
accordingly; every other config field name in the draft was accurate.

**Cron crate specifics verified against the real thing, not assumed:** the draft guessed
`tokio-cron-scheduler = "0.13"`; the version that actually resolves today is `0.15.1`
(confirmed via `cargo add --dry-run`). More importantly, its cron parser is **6-field with a
leading seconds field** (`sec min hour day month weekday[ year]`, confirmed from the crate's
own `Job::new_async` doc example) - not the 5-field `min hour day month weekday` form the
draft's schedules (`"0 2 * * *"` etc.) assumed. Written literally, the draft's schedules would
have silently fired at the wrong times (`"0 2 * * *"` means "every day at 00:02:00", not 2am).
All three real schedules below use the correct 6-field form.

## Sprint 5 — LLM Intelligence (shipped)

**Goal:** Background nightly batch: concept extraction, contradiction detection,
session synthesis, smart promotion scoring. All disabled when `CAIRN_LLM_CONSOLIDATION=false`.

**Corrected against reality before implementing** (same discipline as Sprints 2-4): the draft
got the config reuse call right (`LlmConsolidationConfig` already has `enabled`/`url`/`model`/
`api_key` - just under `CAIRN_LLM_CONSOLIDATION_*` env names, not the draft's guessed
`CAIRN_LLM_*`) but was wrong about several load-bearing details it invented or guessed at:

- **`llm_consolidator::call_llm()` doesn't exist** - the real function is
  `chat_with_config(&LlmConsolidationConfig, prompt: &str) -> Result<String>`
  (`llm_consolidator.rs:129`), synchronous (blocking `ureq`, not `async`), already
  `pub(crate)` for exactly this kind of sibling-module reuse.
- **No JSON-in/JSON-out with the LLM.** Every other LLM interaction in this codebase
  (`LlmConsolidator::parse_facts`/`parse_procedures`/`parse_insights`) asks for lenient custom
  tags (`<fact confidence='0.9'>...</fact>`) and silently drops anything malformed, rather than
  demanding strict JSON that breaks the whole batch on one bad token. `llm_intelligence.rs`
  follows the same convention: `<concept>...</concept>` tags for extraction, a plain YES/NO
  word for contradiction checks, `<summary>...</summary>` for session synthesis.
- **Contradictions are a denormalized field, not a graph RELATE.** `Memory.contradicts:
  Vec<String>` and `Memory.suspicious: bool` already exist (pre-v0.8.0) and are already
  rendered as graph edges by `MemoryEngine::graph()` - there is no `contradicts` relation
  table to `RELATE` into. Flagging a contradiction is a normal `Store::upsert_memory` write
  that appends to the vec and sets `suspicious = true`, the same pattern Sprint 4's
  `run_decay` established for "load, mutate in Rust, write back."
  `/api/capabilities`'s already-hardcoded `contradiction_detection: true` becomes accurate
  for the first time once this ships, rather than describing a capability with no automated
  writer behind it.
- **No `embedding: Vec<f32>` field on `Memory`, and it was never going to get one.** The
  draft's promotion scorer needs `mem.embedding` for the "embedding universality" signal and
  a `global_centroid: Vec<f32>` cache computed by a `global-centroid` cron job. Denormalizing
  the full embedding vector onto every `Memory` row (on top of SurrealDB's own vector index)
  is a real schema change for a nice-to-have third signal - not worth it. **Dropped**: the
  `global-centroid` job and the `univ_score` term entirely. `fast_promotion_score` is
  `kind_prior * 0.4 + cross_score * 0.6` (reweighted from the draft's three-way split now
  that it's two signals, not three).
- **No hand-rolled sensitive-content regex.** `cairn-share`'s `Sanitizer` (used by
  `/api/share/sanitize`) already does exactly this detection - and far more thoroughly
  (API keys, JWTs, Stripe/AWS/GitHub/Slack tokens, emails, IPs, home paths, high-entropy
  blobs) than the draft's three-pattern regex. `run_promotion_scoring` calls
  `cairn_share::Sanitizer::new().sanitize(&mem.content).sensitivity.is_shareable()` instead
  of duplicating weaker detection logic. `cairn-memory` gained a `cairn-share` dependency
  (no cycle - `cairn-share` only depends on `cairn-core`).
- **`kind_prior_score` takes `MemoryKind`, not a string, and drops `"procedural"`.** The
  draft's match arms mixed up `MemoryKind` (`Fact`/`Decision`/`Task`/`Preference`/`Gotcha`/
  `Note`) with `MemoryTier` (`Procedural` is a *tier*, already the query's own tier filter -
  it was never a `kind` value to begin with).
- **No way to read `access_log` back before this sprint - only write to it.** Scoring the
  cross-project signal needed a new `StoreBackend::count_cross_project_access(memory_id,
  exclude_project_id, since) -> Result<i64>` (real impl in `surreal.rs`, no-op-`0` default -
  same hermetic-backend boundary as `record_access_batch`/`prune_access_log_before`).
- **No auto-promotion in Sprint 5.** The draft's Step 7 auto-promotes anything scoring
  `> 0.90`. That's explicitly **Sprint 8's job** ("full-auto promotion + Undo", gated by
  `CAIRN_PROMOTE_THRESHOLD`) per this plan's own autonomy design (see Part B). Sprint 5 only
  ever *writes* `promo_score`; a human calls `/promote` or `/dismiss-promotion` on whatever
  lands in the `[0.70, 0.90]` review band. Building the auto-promote path twice (once here,
  once in S8 with a real threshold and an audit log) would just mean deleting Sprint 5's
  version later.
- **Session synthesis uses headers, not a JSON body.** `RemoteClient` already carries
  `project_id` (Sprint 3) and sends it as `X-Cairn-Project`; Sprint 5 adds a `session_id`
  field the same way, sent as `X-Cairn-Session` - which `scope_middleware` (Sprint 2) already
  parses into `ScopeCtx` server-side. `POST /api/memory/session-summary` needs no request
  body at all; it reads `Extension<ScopeCtx>` like every other scope-aware handler.

## Sprint 6 — RAG Document Store (shipped, auto-sync deferred)

**Goal:** Users can ingest files/URLs as chunked knowledge. Chunks surface in
`cairn assemble` output alongside memories.

**Corrected against reality before implementing** (same discipline as Sprints 2-5, and the
largest gap yet - this sprint invented an entire parallel architecture rather than getting a
few details wrong):

- **Nothing in this codebase is async, including this draft's targets.** The draft's S6.3 and
  S6.5 assume `DocumentEngine`/`cairn-assemble::assemble()`/`MemoryEngine::recall()` are all
  `async fn` reachable via `tokio::join!`. All three are synchronous - confirmed by reading
  `Assembler::assemble()` (`crates/cairn-assemble/src/lib.rs`) and every `MemoryEngine` method
  shipped in Sprints 4-5 this cycle. `cairn-assemble` has zero tokio dependency. Document search
  is a second, sequential, synchronous call, not a parallel one.
- **No raw `store.query(...).bind(...).await` SurrealQL surface** - the fourth sprint in a row
  to assume this. The real pattern (established by `ProjectRecord` in Sprint 3) is: a plain
  struct, a handful of `StoreBackend` trait methods with safe defaults, a real implementation in
  `surreal.rs`, and a real-or-no-op implementation in `memory_backend.rs` depending on whether
  the feature is user-visible (real, here) or pure telemetry (no-op, like `access_log`).
- **No `embedding: Vec<f32>` on the public `Chunk` type** - same call as Sprint 5 declining to
  denormalize embeddings onto `Memory`. Chunk search reuses the exact HNSW mechanism `memory`
  already has (`Store::semantic_recall`'s `<|k,ef|>` operator) against a new `document_chunk`
  table with its own embedding field and HNSW index, entirely inside the store layer - never
  exposed on `DocumentChunkRecord`.
- **No `text-splitter` dependency.** A paragraph-aware chunker (`cairn-document::chunk_text`)
  covers this: pack whole paragraphs up to a character budget, hard-split an over-long
  paragraph on word boundaries, hard-split an over-long single "word" (e.g. a URL) on `char`
  boundaries. Thirteen unit tests, zero new crates.
- **The server cannot read a file path or fetch a URL itself - this is the architectural point
  the draft missed entirely, not just a wrong function signature.** `cairn-server` runs in a
  Docker container (see `crates/cairn-api/src/bin/cairn_server.rs`'s own doc comment); a
  `source` file path in a request body would resolve against the *container's* filesystem, not
  the user's, and having the server fetch arbitrary user-supplied URLs is an SSRF hole for no
  reason. The real design flips this: `cairn-client` (already documented as "a thin HTTP
  wrapper... no local engines" in `main.rs`) reads the file or fetches the URL locally via
  `cairn_document::read_source`, then `POST`s the already-fetched text. The server only chunks
  and stores - it never touches a filesystem path or a third-party URL. `cairn-document` is a
  dependency of *both* `cairn-client` (for `read_source`) and `cairn-api` (for `chunk_text`).
- **No `project_id` scoping in this pass.** The draft's `Document.project_id: Option<String>`
  and `--project <id>` CLI flag would need Sprint 2's `ScopeCtx` threaded through ingest/search/
  list - real work, and not what makes RAG useful day one (a README or design doc is usually
  meant to be visible everywhere, not walled off like a `Project`-scoped memory). Documents ship
  **Global-only**: every ingested chunk is visible to every project/session.
- **`DELETE /api/documents/:id` needs an `id` distinct from `source`** - `source` is a path or
  URL and may contain characters unsafe for a REST path segment (`docs/architecture.md` has a
  `/`). `DocumentSummary.id` is a short content-hash of `source` (same idea as Sprint 3's
  project-id hashing); the delete handler resolves `id` back to `source` via `list_documents()`.
- **S6.3b (auto-sync) is deferred, not shipped.** Gitignore-aware workspace walking needs a new
  dependency (`ignore` or `walkdir` - neither is in the workspace today) plus a `SessionStart`
  hook change plus a new cron job - real, separable work. More importantly, "stop making a human
  remember to run an ingest command" is *exactly* what Sprint 8/9's autonomy charter already
  exists for (see Part B). Auto-sync belongs there, landing on top of this sprint's real ingest/
  chunk/store/search primitives, rather than being bolted onto the sprint that builds those
  primitives. Manual `cairn documents ingest` is the only ingestion path for now.

### What shipped

- **`crates/cairn-document`** (new crate; dependencies `thiserror`, `ureq`, `url` - all
  already-used workspace deps, zero new external crates): `read_source(&str)` (dispatches to
