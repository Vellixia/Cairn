# 32 — v0.8.0: Session TTL Promotion + Cron Job Surface

> **Walked 2026-07-03 against a live `cairn:dev` + `cairn-surreal`. Result: 4/4 steps PASS.**

## Objective
Verify Sprint 4: the in-process cron scheduler exposes the full job list, a `Session`-scoped
memory past `CAIRN_SESSION_TTL_DAYS` promotes to `Global` on `session-gc`, and the promotion is
reflected in both `GET /api/cron/history` and `GET /api/cron/jobs`'s `last_run`.

## Preconditions
- [x] `cairn` :7777 healthy (`cairn:dev`, current source)
- [x] Admin session cookie fresh
- [x] Direct SurrealDB access available for one backdating step (`docker exec cairn-surreal
      /surreal sql ...`) - `POST /api/memory` has no way to set a custom `updated_at`, so
      simulating "N days old" requires writing the timestamp directly. This is the walk's own
      test-setup mechanism, not a server API - noted so a re-run doesn't assume an API for it.

## Surface
API + direct-datastore (test setup only)

## Steps

### Step 1: `GET /api/cron/jobs` lists all 6 jobs with correct schedules
**Do**: `GET /api/cron/jobs` on a freshly booted stack (no jobs have run yet).
**Request**:
```http
GET /api/cron/jobs HTTP/1.1
Cookie: cairn_session=...
```
**Expected**: 6 jobs - `session-gc` (`0 0 2 * * *`), `memory-decay` (`0 0 3 * * Sun`),
`access-log-prune` (`0 0 4 1 * *`), `llm-intelligence` (`0 30 3 * * *`), `memory-demote`
(`0 0 4 * * *`), `tune` (`0 0 5 * * Sun`); all `last_run: null`.
**Observed**: exactly these 6 jobs, exactly these schedules, all `last_run: null`.
**Result**: PASS

### Step 2: Seed a `Session`-scoped memory and force it stale
**Do**: `POST /api/memory` with `X-Cairn-Session: sess-walk32-ttl`, `scope_type: "session"`.
Then, since the API has no field to backdate `updated_at`, directly `UPDATE` the row in
SurrealDB to set `updated_at` 3 days in the past (`CAIRN_SESSION_TTL_DAYS` default is 2).
**Request**:
```http
POST /api/memory HTTP/1.1
X-Cairn-Session: sess-walk32-ttl
{"content": "SESSION-TTL-32: a session-scoped fact that should promote to Global", "kind": "fact", "tier": "episodic", "scope_type": "session"}
```
```sql
UPDATE memory:`<id>` SET updated_at = time::now() - 3d;
```
**Expected**: memory created with `scope_type: "session"`, `scope_id: "sess-walk32-ttl"`; the
SurrealDB update confirms the new `updated_at`.
**Observed**: created as `id=9710bcdf-6889-4609-a4c9-bf4c20186bcf`; SQL update confirmed
`updated_at: d'2026-06-30T04:18:13.497468028Z'` (3 days before the walk's "now").
**Result**: PASS

> **Note on `CAIRN_SESSION_TTL_DAYS=0`:** first attempted this step by setting the TTL to `0`
> and expecting "0 days old" to count as stale immediately. That's wrong -
> `run_session_gc(0)` short-circuits to `Ok(0)` (matching this codebase's `0 = unlimited/off`
> convention everywhere else), so `0` **disables** the job rather than making everything
> instantly eligible. Not a bug - just documenting the correct mental model, since it's an easy
> assumption to get backwards.

### Step 3: `POST /api/cron/run/session-gc` promotes the stale memory
**Do**: manually trigger the job.
**Request**:
```http
POST /api/cron/run/session-gc HTTP/1.1
Cookie: cairn_session=...
```
**Expected**: `{"job": "session-gc", "outcome": "ok", "detail": "promoted 1 session-scoped memories to global"}`
**Observed**: exactly that - `"detail":"promoted 1 session-scoped memories to global"`, `duration_ms: 9`.
**Result**: PASS

### Step 4: Promotion is visible via recall, history, and last_run
**Do**: three follow-up reads.
**Request** (3x):
```http
GET /api/memory/recall?q=SESSION-TTL-32&limit=5 HTTP/1.1     (no scope headers)
GET /api/cron/history?job=session-gc HTTP/1.1
GET /api/cron/jobs HTTP/1.1
```
**Expected**:
- recall with no scope header now returns the memory (it's `Global` now, not `Session`)
- history shows the run with the "promoted 1" detail
- `jobs`'s `session-gc` entry now has a non-null `last_run` matching the history entry
**Observed**:
- recall: 1 result, `scope_type: "global"`, `scope_id: null` - confirms the reassignment, not
  just the job's self-reported count
- history: `[{"job":"session-gc",...,"detail":"promoted 1 session-scoped memories to global"}]`
- jobs: `session-gc.last_run` matches the history entry exactly
**Result**: PASS

## DB Verification
- Step 4's scope-less recall returning the memory with `scope_type: "global"` is itself the DB
  verification - SurrealDB is the only writer, so a correct read after the promotion is
  sufficient proof `reassign_scope` actually committed the change, not just that the job
  reported a count.
