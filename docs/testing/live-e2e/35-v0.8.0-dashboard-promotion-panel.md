# 35 — v0.8.0: Dashboard Promotion & Intelligence Panel

> **Walked 2026-07-03 against the real production build (`cairn-server`'s embedded static
> export, `http://127.0.0.1:7777`) - not the separate `npm run dev` proxy, see the note below.
> Result: 3/3 steps PASS. This closes a gap explicitly left open since Sprint 7 ("not re-verified
> in a live browser") and re-confirmed as a known gap at the end of Sprint 9.**

## Objective
Verify Sprint 7's Promotion & Intelligence dashboard panel (`/memory/promotion`) against a live
server with real seeded data: does it render promotion candidates and the intelligence log
correctly, and does the Promote/Dismiss interaction round-trip to the server?

## Preconditions
- [x] `cairn` :7777 healthy, running current source
- [x] Seeded via API: one `Project`-scoped candidate with `promo_score` set directly to `0.78`
      via SurrealDB (see the note below on why), one manually-promoted memory (populates the
      log), two `llm-intelligence` job runs already in history from doc 33
- [x] Browser viewport explicitly resized to `1280x800` - see the tooling note below

## Surface
browser (via `preview_*` tools, substituting for this repo's usual `chrome-devtools` MCP
convention - not available in this session, but the same rigor: real DOM assertions, no URL
pattern matching)

## Steps

### Step 1: Log in and navigate to `/memory/promotion`
**Do**: fill the login form, submit, navigate to the promotion route.
**Expected**: lands on the dashboard, then the Promotion & Intelligence page renders a heading,
a "Promotion candidates" section, and an "Intelligence log" section.
**Observed**: exactly that. `heading: "Promotion & Intelligence"`, "Promotion candidates" /
"1 candidate(s) . promo_score 0.70-0.90", "Intelligence log" / "Last 10 runs per job, newest
last." with 2 real log entries whose detail text matches doc 33's API responses verbatim
(`"extracted concepts on 1 memories, resolved/flagged 1 contradictions, scored 1 promotion
candidates, auto-promoted 0"`).
**Result**: PASS

### Step 2: Promote candidate round-trips through the real API
**Do**: click the seeded candidate's "Promote" button.
**Expected**: the candidate list count decrements to 0 and the empty-state message appears;
`GET /api/memory/promotion-log` server-side gains a new entry for that memory.
**Observed**: UI updated to `"0 candidate(s) . promo_score 0.70-0.90"` with "Nothing waiting for
review..." Server-side: `promotion-log` gained `{"memory_id":"31400235-...","action":"promote","score":0.78,"reason":"manual",...}` - the click's effect is real, not just an optimistic UI update.
**Result**: PASS

### Step 3: Console is clean
**Do**: `preview_console_logs(level: "error")` after both steps.
**Expected**: empty.
**Observed**: empty.
**Result**: PASS

## Tooling notes (test infrastructure, not product bugs)
- **Viewport was `0x0` initially.** `window.innerWidth`/`innerHeight` read `0` right after
  `preview_start`, which made every element's `getBoundingClientRect()` nonsensical (negative
  coordinates) and caused `preview_click` on the login button to silently click nothing. Fixed
  with an explicit `preview_resize(width: 1280, height: 800)`. Worth remembering for any future
  walk using these tools fresh - don't assume the viewport is sized before the first interaction.
- **`preview_click` stayed unreliable even after the resize fix** - it reported success but
  produced no network activity across several retries, on both the `next dev` server (:3000) and
  the real production build (:7777). Dispatching the click via `preview_eval`'s
  `element.click()` worked immediately and reliably for the rest of the walk (login form, and
  the Promote button in Step 2). Documenting as a tool quirk in this session, not a product bug -
  the same button responds correctly to a genuine click event either way.
- **`preview_screenshot` timed out 3/3 attempts** on this page, with `document.readyState ===
  "complete"` and no console errors. Evidence for this walk rests on the accessibility snapshot
  instead, which this project's own tooling guidance already treats as the preferred signal over
  screenshots for verifying text/structure - not a downgrade in rigor, just no image artifact.
- **Tested against the real production build (:7777), not `cairn-web`'s `npm run dev` (:3000).**
  The dev server logged `Specified "rewrites" will not automatically work with "output: export"`
  and its login form's elements measured at negative/zero-width coordinates in a way the
  production build's did not, even after the same viewport fix. Rather than chase whether that's
  a real dev-mode bug or another tooling artifact, switched to testing what's actually shipped -
  the static export `cairn-server` embeds and serves directly - which is the more meaningful
  target for "does the real product work" anyway.

## DB Verification
- Step 2's server-side `promotion-log` re-read is the DB verification - confirms the click
  produced a real, persisted state change, not just a client-side optimistic update.

## Evidence
- Accessibility snapshots captured inline above (before/after Step 2, showing the candidate
  count and empty-state message change).
- No screenshot image - see tooling note above.

## Findings
- None new. This walk's data was deliberately seeded with a directly-written `promo_score`
  (`0.78`, via SurrealDB) rather than a naturally-scored one, because
  `docs/testing/findings/open/promotion-scoring-ceiling-unreachable.md` (filed in doc 33) means
  no real memory can ever reach the `[0.70, 0.90]` band on its own - without that manual seed,
  this panel would have nothing to show in a real deployment, ever. That's the same finding,
  not a new one; noted here because it directly explains this walk's test setup.

## Walked result
- **Steps walked**: 3/3 PASS. The panel itself - rendering, live data, and the Promote
  interaction's full round-trip to the server - all work correctly. The panel's real-world
  emptiness is a consequence of the scoring-ceiling finding, not a dashboard bug.
