# 33 — v0.8.0: LLM Intelligence (Concept Extraction, Contradiction Detection, Promotion Tie-Break)

> **Walked 2026-07-03 against a live `cairn:dev` + a throwaway fake OpenAI-compatible stub
> (`cairn-fake-llm`, not part of the shipped product). Result: 2/3 mechanisms PASS live; 1
> confirmed unreachable in practice (see Findings) - not a crash, a structural dead path.**

## Objective
Verify Sprint 5's three `llm-intelligence` cron job phases actually call an LLM and use its
response when `CAIRN_LLM_CONSOLIDATION=true`: concept extraction, contradiction detection, and
promotion-scoring's LLM tie-break for borderline `fast_score`s.

## Preconditions
- [x] `cairn` :7777 healthy, `CAIRN_LLM_CONSOLIDATION=true`,
      `CAIRN_LLM_CONSOLIDATION_URL=http://cairn-fake-llm:8080/v1/chat/completions`
- [x] `cairn-fake-llm`: a throwaway Python stdlib HTTP server on the `cairn_default` network,
      answering `/v1/chat/completions` with a canned response chosen by matching keywords in the
      prompt (concept-extraction tags, YES/NO for a planted contradiction marker, UNIVERSAL for
      the promotion tie-break, each with `usage.total_tokens`). Confirmed reachable from `cairn`
      by service name before this walk began. **Not real intelligence** - a fixed responder, only
      useful for proving the *wiring* (request sent, response parsed, result applied), not
      response quality.
- [x] Admin session cookie fresh

## Surface
API (cron job) + a throwaway fake LLM

## Steps

### Step 1: Concept extraction populates `concepts` on a concept-less memory
**Do**: `POST /api/memory` with no `concepts`, then `POST /api/cron/run/llm-intelligence`.
**Request**:
```http
POST /api/memory HTTP/1.1
{"content": "CONCEPT-33: Docker containers provide isolated environments for live testing SurrealDB deployments.", "kind": "note", "tier": "working"}

POST /api/cron/run/llm-intelligence HTTP/1.1
```
**Expected**: the memory's `concepts` field is populated from the stub's canned
`<concept>...</concept>` response after the job runs.
**Observed**: `"concepts":["docker","live-testing","surrealdb"]` - exactly the stub's canned
concept-extraction response. Job detail: `"extracted concepts on 6 memories, ..."` (6, not 1 -
every concept-less memory left over from docs 31/32 got filled in the same pass, which is
correct: the job scans the whole corpus, not just new arrivals).
**Result**: PASS

### Step 2: Contradiction detection calls the LLM and records a `contradicts` edge
**Do**: seed two memories with overlapping wording (so the hashing embedder's nearest-neighbor
search finds them as candidates) - only one carries a marker word the stub reads as "these
contradict."
**Request**:
```http
POST /api/memory  {"content": "CONTRA-33: the walk32 test service listens on port 9001 for incoming requests", ...}
POST /api/memory  {"content": "CONTRA-33: ...port 9002... e2e-contradiction-marker", ...}
POST /api/cron/run/llm-intelligence
```
**Expected**: the marker-bearing memory's `contradicts` includes the other's id; both become
`suspicious: true` (or one auto-resolves via `supersedes` if confidence/recency differ - here
they don't, both start at confidence 0.5, so the flag-both path applies, per Sprint 8's design).
**Observed**: `dc3b4afa...` (marker) has `"contradicts":["7933d9c6-...(the non-marker one)"],
"suspicious":true"`. Confirmed calling out to the stub and acting on "YES".
**Result**: PASS, **with a test-design caveat**: the stub matches on the *combined* prompt
(both statements), so the marker-bearing memory also got flagged against a couple of unrelated
memories left over from docs 31/32 that happened to be semantic neighbors (any comparison
involving it always reads as "contradicts" to this simplistic stub). That's an artifact of a
keyword-matching fake LLM, not a product bug - a real LLM wouldn't do this. The one deliberately
planted pair resolved correctly regardless.

### Step 3: Promotion scoring's LLM tie-break — does it ever actually fire?
**Do**: seed a `Project`-scoped `Semantic`-tier `Fact` (the only kind/tier/scope combination
`run_promotion_scoring` considers), run `llm-intelligence`, and check whether the resulting
`promo_score` reflects any LLM blending.
**Request**:
```http
POST /api/memory HTTP/1.1
X-Cairn-Project: proj-promo-33
{"content": "PROMO-33: rust ownership prevents use-after-free bugs at compile time", "kind": "fact", "tier": "semantic", "scope_type": "project"}

POST /api/cron/run/llm-intelligence HTTP/1.1
```
**Expected** (per the source): `fast_promotion_score(Fact, cross_hits=0) = 0.9*0.4 + 0*0.6 =
0.36`. Since `0.36` is below the `[0.40, 0.85]` LLM-tie-break band, the LLM branch should be
skipped and `promo_score` should land at exactly `0.36`, unmodified.
**Observed**: `"promo_score":0.35999998` - exactly the pure `fast_score`, confirming the LLM
branch did not fire for this memory.
**Result**: PASS as far as it goes, **but this raises a structural question investigated next**:
is `[0.40, 0.85]` ever actually reachable in production, or is the LLM tie-break dead code?

## DB Verification

- Step 3's `promo_score` read the same way.

## Findings

### Finding: promo_score can never exceed 0.36 - the LLM tie-break is one symptom of a wider ceiling

Filed as `docs/testing/findings/open/promotion-scoring-ceiling-unreachable.md` (see that file for
the full writeup - it turned out to be bigger than just this one branch). Summary:
`fast_promotion_score`'s only way to reach the `[0.40, 0.85]` LLM-tie-break band is a nonzero
`cross_project_hits`, which comes from `count_cross_project_access` counting `access_log` rows
logged by a *different* project than the memory's own. But `access_log` rows are only ever
written for memories `recall_for_org` actually returns, and its own scope filter requires
`m.scope_id == scope.project_id` for `Project`-scoped memories - the exact condition that makes
"a different project accessed it" impossible to log in the first place. Verified empirically in
Step 3: the observed `promo_score` (0.35999998) is bit-for-bit the pure `fast_score`, which is
only possible if the LLM blend (`score*0.5 + llm_score*0.5`) never ran.

Following this further (see the filed finding): `0.36` isn't just below the LLM tie-break band -
it's below *every* threshold in the promotion pipeline, including the `[0.70, 0.90]`
human-review band and the `0.85` full-auto-promote default. This isn't a narrow LLM-wiring gap;
it means the promotion pipeline's automatic discovery mechanism (Sprint 5) and full-auto
promotion (Sprint 8) never fire against real data at all, not just the LLM refinement step.
