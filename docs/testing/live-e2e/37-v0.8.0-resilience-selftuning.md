# 37 — v0.8.0: Resilience & Self-Tuning (Offline Spool, Cron Health, LLM Budget, Tune, Hygiene)

> **Walked 2026-07-03 against a live `cairn:dev`. Result: 6/6 mechanisms PASS live. This is
> Sprint 9's own final verification note calling live end-to-end testing a known gap - this walk
> closes it.**

## Objective
Verify every Sprint 9 resilience mechanism against a live stack: the offline hook spool
(queue -> replay -> drain), cron health reporting, the LLM daily token budget (per-item
exhaustion mid-batch), the `tune` self-tuning job (minimum-sample gate + real threshold effect),
and memory hygiene (dedup sweep + per-project working-tier cap).

## Preconditions
- [x] `cairn` :7777 healthy; `cairn-client` built fresh in `cairn-rustbuild` (joined to
      `cairn_default`)
- [x] Fake LLM stub reachable - **re-confirmed live immediately before use**, since its
      `docker exec`-launched process was found dead partway through this session (see Findings;
      a test-infrastructure issue, not a product bug)
- [x] Several `.env` overrides applied one at a time, each requiring a `docker compose up -d`
      restart to take effect: `CAIRN_LLM_DAILY_BUDGET=500`, then
      `CAIRN_MAX_WORKING_PER_PROJECT=3` - reverted to checked-in defaults after this walk

## Surface
CLI/hook + API + direct-datastore (test setup only)

## Steps

### Step 1: Offline spool — queue, replay, and land for real
**Do**: `docker compose stop cairn`; run `cairn hook UserPromptSubmit` (built binary, pointed at
`http://cairn:7777`) with a distinctive prompt; confirm `~/.cairn/spool.jsonl` gained lines;
`docker compose start cairn`; run `cairn hook SessionStart`; confirm the spool drains **and** the
prompt is actually retrievable from the server, not just "the file emptied."
**Expected**: 2 spool lines while offline (`/api/guard/anchor/auto`, `/api/memory`), each with
correct `project_id`/`session_id`; both delivered on the next `SessionStart`; spool file gone
afterward; the prompt content shows up in a real memory.
**Observed**:
- While offline: exactly 2 JSONL lines, e.g.
  `{"path":"/api/memory","body":{"content":"OFFLINE-SPOOL-37: ...","kind":"note","tier":"episodic"},"project_id":"ca3a4e6ad7b8c8ce","session_id":"walk-session-37-offline","ts":"..."}`
- `SessionStart`'s own `additionalContext` output afterward included
  `"- (note) OFFLINE-SPOOL-37: this prompt should be spooled since the server is down"` in its
  wakeup list - the spooled memory didn't just "get accepted," it's live and recallable.
- `cat ~/.cairn/spool.jsonl` afterward: `No such file or directory` - fully drained.
**Result**: PASS

### Step 2: `GET /api/cron/health` reflects real run state
**Do**: check health on a fresh boot, then after manually triggering `session-gc`.
**Expected**: fresh boot shows all 6 jobs `running: false`, `last_run_at: null`, `stale: false`;
after a trigger, that job's `last_run_at`/`last_status` populate and match
`GET /api/cron/history`.
**Observed**: exactly that in both cases.
**Result**: PASS. **Single-flight itself was not raced live** - sequentially triggering the same
job twice never overlaps since `run_job_now` is synchronous; genuinely racing it would need two
concurrent processes hitting the manual-trigger endpoint at the exact same instant, which isn't
practical to construct through the HTTP API alone. Covered instead by
`crates/cairn-api/src/cron.rs`'s own unit tests (`single_flight_guard_blocks_a_second_start...`,
`run_job_now_skips_and_records_when_already_running`), which exercise the guard directly.

### Step 3: LLM daily budget stops a batch partway, precisely
**Do**: restart with `CAIRN_LLM_DAILY_BUDGET=500` (the fake stub reports exactly 500
`total_tokens` per call). Seed 3 concept-less memories, trigger `llm-intelligence`.
**Expected**: the first call succeeds and consumes the entire budget; the loop stops before the
2nd and 3rd items; job detail reflects a partial count, not 0 or 3.
**Observed**: `"extracted concepts on 1 memories, ..."`. Direct check of all 3 seeded memories:
exactly one (`item-3`) gained `concepts: ["docker","live-testing","surrealdb"]`; the other two
remained `concepts: []`, untouched.
**Result**: PASS - the budget guard is real and per-item, not an all-or-nothing gate.

### Step 4: `tune` job — minimum-sample gate, then a real threshold change
**Do**: trigger `tune` immediately (few queries observed this session); then issue 20 distinct
`recall` queries (deliberately non-repeating, so `followup_rate` stays at `0.0`); trigger `tune`
again.
**Expected**: first call skips ("too few queries"); second call adjusts `promote_threshold`
downward by the `0.05` step (low followup rate reads as "recall is working well, ease up").
**Observed**: `"skipped - only 3 queries observed, want at least 20"`, then
`"followup_rate=0.000 over 23 queries, promote_threshold 0.85 -> 0.80"`.
**Result**: PASS

### Step 5: The tuned threshold is actually consulted, not just computed
**Do**: seed a `Working`-tier `Project`-scoped memory (invisible to `run_promotion_scoring`'s
tier filter, so its manually-set score survives) with `promo_score = 0.82` - below the *default*
`0.85` but above the *tuned* `0.80` from Step 4. Trigger `llm-intelligence`.
**Expected**: it gets auto-promoted, proving `run_auto_promote`'s caller reads
`effective_promote_threshold` (the live override), not the static config default.
**Observed**: job detail included `"auto-promoted 1"`; the memory's state afterward:
`scope_type: "global", promo_locked: true, promo_score: 0.82` - promoted at a score that would
have been rejected under the unmodified default.
**Result**: PASS - this is the clearest possible evidence the self-tuner isn't cosmetic.

### Step 6: Memory hygiene — dedup sweep and working-tier cap
**Do (dedup)**: create two distinct memories, then `POST /api/memory/:id` to edit one's content
to exactly match the other's (the realistic convergence scenario, not insert-time duplication -
`POST /api/memory` itself already dedups identical content at creation, confirmed when two
identical inserts returned the *same* id). Trigger `memory-decay`.
**Do (cap)**: restart with `CAIRN_MAX_WORKING_PER_PROJECT=3`; create 5 `Working`-tier
`Project`-scoped memories one second apart under one project. Trigger `memory-decay`.
**Expected (dedup)**: the older of the converged pair is deleted, the newer survives.
**Expected (cap)**: the 2 oldest of the 5 are deleted, the 3 newest survive.
**Observed**:
- Dedup: job detail `"...deduped 1..."`. The older memory: `DELETE` -> `{"error":"no such
  memory","error_code":"not_found"}`. The newer (post-edit) memory: still present with the
  converged content.
- Cap: job detail `"...capped 2 over-quota working memories"`. Deleting all 5 by id afterward:
  the 2 oldest -> `not_found` (already gone); the 3 newest -> `{"deleted":true}` (still existed
  until this check deleted them).
**Result**: PASS on both - exact counts, exact survivors, in both cases.

## DB Verification
- Every step re-reads state after acting (recall, delete-as-existence-check, cron
  history/health) rather than trusting a job's self-reported count alone - consistent with every
  other doc in this series, and especially important here since Sprint 9's own code comments
  describe every one of these mechanisms as "best-effort."

## Findings

### Test-infrastructure note (not a product bug): the throwaway fake LLM stub died mid-session
The `cairn-fake-llm` container (a throwaway Python stdlib HTTP server, not part of the shipped
product - see doc 33) was found with an uptime far shorter than expected partway through this
walk, and its `docker exec`-launched Python process was gone entirely (only the container's own
`sleep infinity` PID 1 remained). This silently caused `llm-intelligence` runs in between to
report "extracted concepts on 0 memories" for reasons unrelated to any Sprint 9 feature - the
LLM calls were failing to connect, and every LLM-dependent function's own error handling
(`Vec::new()`/`false` on failure) absorbed it silently, exactly as designed for a real outage.
Caught by checking `updated_at` timestamps not advancing across multiple runs where they should
have. Fixed by restarting the stub process and re-confirming reachability before Step 3. No
product code is implicated - this was a limitation of the ad hoc test double, not of Cairn.

### No new product findings this walk
Every mechanism tested behaved exactly as designed, with the precision to prove it (exact
counts, exact survivors, exact threshold arithmetic) rather than just "something happened."
