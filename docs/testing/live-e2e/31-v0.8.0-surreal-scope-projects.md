# 31 — v0.8.0: SurrealDB Backend, Scope Model, Auto Project Detection

> **Walked 2026-07-03 against a freshly rebuilt live `cairn:dev` (current source, all 9 v0.8.0
> sprints) + `cairn-surreal` (SurrealDB v3). Result: 2/3 areas PASS, 1 confirmed real bug (see
> Findings).**

## Objective
Verify three foundational v0.8.0 sprints against a live stack: Sprint 1 (SurrealDB replaces
HelixDB+MinIO), Sprint 2 (scope model isolates recall by `X-Cairn-Project`), Sprint 3 (the
`cairn` CLI's `hook SessionStart` auto-detects a project and registers it server-side).

## Preconditions
- [x] `cairn` :7777 healthy, running `cairn:dev` (built from current source, not the stale
      `cairn:local` image that predates this plan)
- [x] `cairn-surreal` healthy (internal only, no host port - confirmed by design)
- [x] Admin session cookie fresh (`/tmp/walk/cookies.txt`, `POST /api/auth/setup` + `/login`)
- [x] `cairn-client` built fresh inside `cairn-rustbuild` (network-joined to `cairn_default`)
      so `cairn hook` can reach the live server by its internal DNS name (`http://cairn:7777`)
- [x] A `write`-scope device token minted via `POST /api/devices/tokens` for the CLI

## Surface
combined: API + CLI/hook

### Step 3: Scope isolation — remember under two different projects
**Do**: `POST /api/memory` twice, once with `X-Cairn-Project: proj-alpha-walk31`, once with
`X-Cairn-Project: proj-beta-walk31`, each a `Project`-scoped fact with a unique marker string.
**Request** (2x):
```http
POST /api/memory HTTP/1.1
Cookie: cairn_session=...
X-Cairn-Project: proj-alpha-walk31
{"content": "ALPHA-SCOPE-31: secret alpha project fact about widgets", "kind": "fact", "tier": "working", "scope_type": "project"}
```
(and the same shape for beta, `X-Cairn-Project: proj-beta-walk31`, content `BETA-SCOPE-31: ...`)
**Expected**:
- 2x 200
- Each response's `scope_type: "project"`, `scope_id` equal to the project header sent
**Observed**:
- alpha: `id=135180f4-915e-4459-ac3c-2d633802cf92`, `scope_type=project`, `scope_id=proj-alpha-walk31`
- beta: `id=98fc7b27-5a13-4ec9-ba47-f760b1588266`, `scope_type=project`, `scope_id=proj-beta-walk31`
**Result**: PASS

### Step 4: Scope isolation — recall respects the caller's project header
**Do**: `GET /api/memory/recall?q=SCOPE-31` three times: once as alpha, once as beta, once with
no project header at all.
**Request** (3x):
```http
GET /api/memory/recall?q=SCOPE-31&limit=10 HTTP/1.1
Cookie: cairn_session=...
X-Cairn-Project: proj-alpha-walk31   (omitted on the 3rd call)
```
**Expected**:
- As alpha: only the ALPHA-SCOPE-31 memory returned
- As beta: only the BETA-SCOPE-31 memory returned
- With no project header: neither returned (both are `Project`-scoped, not `Global`)
**Observed**:
- As alpha: 1 result, `id=135180f4...` (ALPHA), score 0.0200
- As beta: 1 result, `id=98fc7b27...` (BETA), score 0.0198
- No header: `[]`
**Result**: PASS - clean isolation in both directions, and the no-header case confirms
`Project`-scoped memories don't leak into a scope-less view either.

### Step 5: `cairn hook SessionStart` auto-detects a project
**Do**: in a freshly `git init`-ed directory (`/tmp/test-project-alpha`, no remote, no commits),
with `CAIRN_SERVER=http://cairn:7777` and `CAIRN_TOKEN=<device token>` set, pipe
`{"session_id":"walk-session-31"}` into the built `cairn` binary's `hook SessionStart`.
**Request** (what the hook is expected to send):
```http
PATCH /api/projects/upsert HTTP/1.1
Authorization: Bearer <device token>
{"id": "<sha256-derived hash of /tmp/test-project-alpha>", "name": "test-project-alpha", "path": "/tmp/test-project-alpha"}
```
**Expected**:
- The hook process exits 0 (per its own "never break the agent" contract) - not itself proof of success
- `GET /api/projects` subsequently includes a project named `test-project-alpha`
**Observed**:
- Hook process ran, exited 0, printed nothing to stdout/stderr (expected - the digest/anchor/wakeup
  content is empty on a fresh server, so `SessionStart` legitimately has nothing to emit)
- `GET /api/projects` afterward: `[]` for the hook's own attempt (a manually-issued
  `PATCH /api/projects/upsert` with a different id, done as a control, *did* show up correctly -
  see Findings)
**Result**: **FAIL** - see Findings below for root cause.

## Findings

### Finding: `cairn hook SessionStart` never registers the auto-detected project (wrong HTTP method)

**Severity**: high (the entire Sprint 3 auto-detection feature is a no-op in production; scope
isolation itself - Sprint 2 - is unaffected and works correctly independent of this)
**Discovered**: 2026-07-03, this walk, Step 5

**What happened**: `crates/cairn-client/src/hook.rs`'s `SessionStart` handler calls
`rc.post_spooled("/api/projects/upsert", ...)`, which sends an HTTP **POST**
(`RemoteClient::post` wraps `ureq::post(...)`). The server's route is registered as
`.route("/api/projects/upsert", patch(upsert_project))` in `crates/cairn-api/src/lib.rs` -
**PATCH only**. Every call gets a `405 Method Not Allowed`, which isn't a
`ureq::Error::Transport`, so Sprint 9's offline-spool logic correctly treats it as "the server
answered, nothing to retry" and silently drops it - exactly as designed for a genuine HTTP
error, just on the wrong kind of failure here. Confirmed live: `curl -X POST
.../api/projects/upsert` returns `405`; the identical body via `curl -X PATCH` returns `200`
and the project appears in `GET /api/projects` immediately.

**Root cause confirmed via `git log -S`**: this has been broken since the feature was
introduced - `b7024a2` ("feat(cairn-client): auto project detection + registry (v0.8.0 Sprint
3)") already used `.post(...)` for this call. It predates Sprint 9's spool work entirely (spool
only changed *what happens on a transport failure*, not the HTTP method used).

**Expected**: after `cairn hook SessionStart` runs once in a git-initialized directory,
`GET /api/projects` includes an entry for it.

**Actual**: `GET /api/projects` never gains an entry from the hook, in any of the sessions
tested. Auto project detection - the entire point of Sprint 3 - has silently never worked
end-to-end since it shipped, only appearing to work in `crates/cairn-tests`' hermetic HTTP-layer
tests because those call the server's `upsert_project` handler directly (or via the correct
PATCH method), never through the real CLI client.

**Suggested fix**: `crates/cairn-client/src/hook.rs` needs a `patch()` counterpart to its
existing `get()`/`post()` methods (`ureq::patch(...)`, same `with_scope_headers` wrapping), and
the `SessionStart` project-upsert call site should use it (ideally via a `patch_spooled`
sibling to `post_spooled`, so the Sprint 9 offline-spool protection isn't lost for this call).
