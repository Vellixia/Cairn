# 36 — v0.8.0: Autopilot (Promotion, Drift, Auto-Anchor, Digest)

> **Walked 2026-07-03 against a live `cairn:dev`. Result: 4/4 mechanisms PASS live (one via a
> deliberate test workaround - see the note on the promotion-scoring ceiling finding).**

## Objective
Verify Sprint 8: full-auto promotion + `/demote` undo, the drift autopilot policy matrix
(`ok`/`warn`/`danger` × `safe`/`off`/`all`), auto-anchor derivation, and the autopilot digest.

## Preconditions
- [x] `cairn` :7777 healthy
- [x] Baselines established via `GET /api/context/read` before each `verify` call - `verify_edit`
      compares against a previously-read baseline, not a live disk read; skipping this step
      makes every edit look like "new file" (`Risk::Ok`) regardless of content, which is a test
      setup detail worth remembering, not a bug (see Findings).
- [x] Real files under the mounted (read-only) `/workspace` used as edit targets: one under
      `docs/**` (a safe glob) and one under `crates/**` (not a safe glob)

## Surface
API (direct-datastore for one test-setup step, same convention as doc 32)

## Steps

### Step 1: Drift autopilot matrix under default `CAIRN_DRIFT_AUTOPILOT=safe`
**Do**: three `POST /api/guard/verify` calls against real files, each preceded by a fresh
`GET /api/context/read` baseline:
1. `docs/testing/live-e2e/32-...md` (safe glob), edited to remove 29% of its lines -> `warn`
2. same file, edited to remove 93% of its lines with no replacement -> `danger`
3. `crates/cairn-core/src/config.rs` (not a safe glob), edited to remove 28% -> `warn`
**Expected**: (1) auto-approved (`warn` + safe glob), (2) stays `pending` (`danger` never
auto-approves, even under a safe glob), (3) stays `pending` (`warn`, but not a safe glob).
**Observed**:
1. `{"risk":"warn",...}`, drift log: `"status":"approved"`, detail `"...(auto-approved:
   autopilot:safe-path)"`
2. `{"risk":"danger",...}`, drift log: `"status":"pending"` - no auto-approve note
3. `{"risk":"warn",...}`, drift log: `"status":"pending"` - no auto-approve note
**Result**: PASS

### Step 2: `danger` never auto-approves even under `CAIRN_DRIFT_AUTOPILOT=all`
**Do**: restart with `CAIRN_DRIFT_AUTOPILOT=all`, re-establish baselines, repeat the `warn` edit
on the non-safe-glob file and the `danger` edit on the safe-glob file.
**Expected**: the `warn` edit now auto-approves (any path, under `all`); the `danger` edit still
never does.
**Observed**: `crates/cairn-core/src/config.rs` warn -> `"status":"approved"`, detail
`"...(auto-approved: autopilot:all)"`. The `danger` edit -> still `"status":"pending"`.
**Result**: PASS - confirms "danger always holds regardless of mode" is real, not just true for
the default mode.

> **Not re-tested live**: `CAIRN_DRIFT_AUTOPILOT=off` (nothing auto-approves, full manual
> review) and the plain `ok`-risk no-drift-event case. Both are simple, already covered by
> `cairn-guard`'s own policy-matrix unit tests and
> `crates/cairn-tests/tests/29_http_api_autopilot.rs`'s hermetic coverage of all four modes -
> re-running them live wouldn't add confidence proportional to the extra restart cycles.

### Step 3: Auto-anchor derives once, never re-derives
**Do**: `POST /api/guard/anchor/auto` twice with different prompts, on a fresh (never-anchored)
server.
**Expected**: first call derives an anchor from the prompt (`derived: true`); second call leaves
it unchanged (`derived: false`).
**Observed**: first call -> `{"anchor":"Fix the flaky login test in the auth module before the
release","derived":true,...}`. Second call (unrelated prompt) -> `{"anchor":"Fix the flaky login
test...", "derived":false}` - the original anchor is untouched.
**Result**: PASS

### Step 4: Full-auto promotion + `/demote` undo
**Do**: this needed a deliberate workaround - see the Findings note below on why a real
memory's `promo_score` can never organically reach the `0.85` default threshold.
`run_promotion_scoring` only scores `Semantic`/`Procedural`-tier memories, but
`run_auto_promote` has no tier restriction at all - so a `Working`-tier `Project`-scoped memory
with its `promo_score` set directly (via SurrealDB) is invisible to scoring (wrong tier, never
gets overwritten) but *is* visible to auto-promote. Seeded one at `0.90`, triggered
`llm-intelligence`, then called `/demote`.
**Request**:
```http
POST /api/cron/run/llm-intelligence HTTP/1.1
POST /api/memory/<id>/demote HTTP/1.1
```
**Expected**: the job's `detail` shows `auto-promoted 1`; the memory becomes `scope_type:
"global"`, `promo_locked: true`; the promotion log shows `reason: "auto-threshold"`; `/demote`
reverts it to its original `scope_type: "project"`, `scope_id`.
**Observed**: job detail: `"...auto-promoted 1"`. Memory after: `scope_type: "global",
scope_id: null, promo_locked: true`. Promotion log: `{"action":"promote","score":0.9,"reason":
"auto-threshold",...}`. After `/demote`: `scope_type: "project", scope_id:
"proj-autopromote-36"` - exactly its original scope.
**Result**: PASS (mechanism confirmed real and correct; the score that triggers it can't occur
organically today - see Findings)

### Step 5: Autopilot digest reflects activity
**Do**: `GET /api/memory/autopilot-digest?hours=24` after the above steps.
**Expected**: nonzero `promoted`/`drift_auto_approved` counts.
**Observed**: `{"promoted":2,"demoted":0,"drift_auto_approved":1}` - `drift_auto_approved` shows
1, not 2, because the container restart between Step 1 and Step 2 reset the in-process drift-event
log (confirmed separately: drift event `id` numbering restarted from 1 after the restart) - the
digest is counting real, current data, just from a shorter window than the whole walk. Not a bug.
**Result**: PASS

## DB Verification
- Every step re-read the affected resource after acting (drift log, recall, promotion log)
  rather than trusting a single response body - consistent with every other doc in this series.

## Findings
- **Reuses** `docs/testing/findings/open/promotion-scoring-ceiling-unreachable.md` (filed in doc
  33): Step 4 needed a tier-filter workaround specifically *because* of that finding - a real
  `Semantic`/`Procedural` memory can never reach `0.85` on its own, so demonstrating
  `run_auto_promote`'s mechanism at all required sidestepping `run_promotion_scoring` entirely.
  No new finding filed here; this is corroborating evidence for the existing one, from the
  autopilot angle rather than the scoring angle.

## Walked result
- **Steps walked**: 5/5 PASS (4 mechanisms - drift matrix, auto-anchor, auto-promote/demote,
  digest - all confirmed correct against live behavior)
- Drift autopilot and auto-anchor work exactly as designed with no caveats. Full-auto promotion's
  *mechanism* (score above threshold -> promote -> logged -> demotable) is correct, but - per the
  Sprint 9 Definition-of-Done note this walk reinforces - it will never fire in a real deployment
  without the scoring-ceiling finding being addressed first.
